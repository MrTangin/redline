# Modular Toggle Architecture

This template's core promise is: a disabled module is **absent**, not hidden. No package in `package.json`, no config file on disk, no provider mounted, no env var required. This skill documents the mechanics that make that true, so a new module added later follows the same pattern instead of drifting.

## The four pieces

1. **`lib/project.config.ts`** — the single source of truth for on/off state (`projectConfig.modules.<name>: boolean`). Nothing outside this file hardcodes a module's state. It's written by `scripts/setup.mjs` and `scripts/modules/add-*.mjs`, not hand-edited in normal use.
2. **`lib/env.ts`** — one Zod schema per module. `readModuleEnv()` only ever validates the schema for a module that's enabled — a disabled module's env vars are simply never checked, never required. `validateEnv()` is called once from `instrumentation.ts`'s `register()` hook, so a missing var throws a clear, named error at server startup, never a silent half-broken state (see §0 "fail loudly, never silently" in the build spec).
3. **`scripts/modules/add-<module>.mjs`** — the only place a module's packages get installed and its files get scaffolded. Each one:
   - `pnpm add`s exactly that module's packages (nothing else)
   - writes that module's dedicated files (e.g. `server/db/client.ts`, `proxy.ts`, `sentry.server.config.ts`)
   - prompts for and writes real env values to `.env.local`
   - flips its flag in `lib/project.config.ts` via `readProjectConfig()`/`writeProjectConfig()`
   - regenerates any shared generated file it touches (see below)
   
   Runnable standalone (`pnpm run add:auth`) or imported and called by `scripts/setup.mjs`. There is deliberately no `remove-<module>.mjs` — turning a module back off is a manual revert (delete its files, flip its flag, run `pnpm remove <packages>`), since safely reversing scaffolded app code isn't something a script can do blindly.
4. **`scripts/setup.mjs`** — the first-run wizard. Asks yes/no per module, and for every "yes" imports and calls that module's `install()`. Answering "no" does nothing at all — that's what guarantees a from-scratch "everything off" clone has zero optional dependencies.

## The regenerated-file pattern

Some files need to reflect *every currently-enabled module at once* — you can't just append to them per-module, because a static top-level `import` of an uninstalled package breaks the build even behind a runtime `if`. Bundlers resolve imports statically; `if (moduleEnabled) import { Foo } from "not-installed-package"` still fails to compile when the package is absent.

The fix used here: two files are fully **regenerated from scratch** every time any module's install script runs, built from the current `modules` state rather than hand-edited incrementally:

- **`components/providers/providers.tsx`** — regenerated by `regenerateProvidersFile(modules)` in `scripts/modules/lib.mjs`. Composes only the providers for currently-enabled modules (Clerk, PostHog) into nested JSX. Zero providers enabled → it's a passthrough `<>{children}</>`.
- **`instrumentation.ts`** — regenerated by `regenerateInstrumentationFile(modules)`. Always calls `validateEnv()`; additionally imports and boots Sentry's server/edge config only when `monitoring` is enabled.

If you add a fifth module that needs to contribute to one of these shared files, extend the corresponding `regenerate*File()` function in `scripts/modules/lib.mjs` rather than hand-splicing the generated file — the next module script run will otherwise clobber your edit.

Files that are inherently per-module (e.g. `server/db/client.ts`, `sentry.server.config.ts`) don't need this treatment — they either exist (module on) or don't (module off), no composition required.

## Adding a fifth module

1. Add its schema to `moduleSchemas` in `lib/env.ts` and its `false` flag to `projectConfig.modules` in `lib/project.config.ts`.
2. Write `scripts/modules/add-<name>.mjs` following the shape of the existing four: `pnpmAdd(...)`, `writeFile(...)` for its dedicated files, prompt + `upsertEnvLocal(...)`, `readProjectConfig()`/`writeProjectConfig()`, and call the relevant `regenerate*File()` if it needs to contribute to a shared file.
3. Register it in the `MODULES` array in `scripts/setup.mjs` and add an `add:<name>` script to `package.json`.
4. Test both directions in isolation: a fresh clone with it enabled builds and runs; a fresh clone with it left disabled has zero trace of its package or files.

## Common mistake to avoid

Don't read `NEXT_PUBLIC_*` vars through `lib/env.ts`'s `getXEnv()` helpers from a `"use client"` component — those helpers call `schema.safeParse(process.env)` against the *whole* env object, which only resolves server-side. In client components, reference `process.env.NEXT_PUBLIC_FOO` as a literal static expression instead (see `components/providers/posthog-provider.tsx`) so Next.js's bundler can statically inline it. `lib/env.ts` is for server-side startup validation and server-only reads (e.g. `CLERK_SECRET_KEY`, `SUPABASE_SERVICE_ROLE_KEY`), not for client bundles.
