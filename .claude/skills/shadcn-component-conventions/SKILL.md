# shadcn/ui Component Conventions

shadcn/ui components are copied into this repo, not installed as an npm dependency — every file under `components/ui/` is source you own and can edit directly. That means "where does this go" and "do I edit the primitive or wrap it" are decisions this repo needs to be consistent about.

## Directory layout

- **`components/ui/`** — shadcn primitives only, one file per component, added via the CLI (`pnpm dlx shadcn@latest add <component>`). Don't hand-write new files here that didn't come from the CLI — if you need a shadcn-style primitive that doesn't exist in the registry, build it under `components/` (not `components/ui/`) so it's clear it's custom.
- **`components/sections/`** — page-level section components (hero, tech-stack grid, FAQ, etc.), currently used by the showcase page (`app/page.tsx`). One file per section, named after what it shows (`hero.tsx`, `tech-stack-grid.tsx`), composed together in `app/page.tsx`. Sections compose primitives from `components/ui/` — they don't redefine button/card styles inline.
- **`components/providers/`** — module provider components, several of which are generated (see [[modular-toggle-architecture]]). `providers.tsx` is the aggregator regenerated by the module scripts; don't hand-edit its wrapping. Module-specific provider files (`posthog-provider.tsx`) are hand-written templates emitted by their `add-*.mjs` script.
- **`lib/utils.ts`** — the `cn()` helper (clsx + tailwind-merge), written once by `shadcn init`. Use it for every conditional className, don't reimplement className merging elsewhere.

## Installing a new primitive

Run `pnpm dlx shadcn@latest add <component>` rather than hand-copying from the shadcn docs — the CLI resolves the current registry version, applies this project's `components.json` config (style, base color, aliases), and updates `components/ui/` and `lib/utils.ts` consistently. Install components on demand as pages need them, not preemptively — an unused primitive sitting in `components/ui/` is dead weight this template's own "no dead dependencies" philosophy argues against.

## Extending a primitive vs. writing a new component

- **Extend the primitive in place** (edit the file under `components/ui/`) when the change is a genuine variant of the same component — a new `size` or `variant` option on `Button`, an additional color option on `Badge`. These go in the component's `cva()` variants object, following the existing variant/size key naming already in the file (don't invent a third axis unless the design actually needs one).
- **Write a new component** (in `components/` or `components/sections/`, not `components/ui/`) when you're composing multiple primitives into something with its own behavior — a stat card that's a `Card` + custom layout + specific typography, a module toggle switch used in the "Modular by Design" showcase section. These import from `components/ui/` rather than duplicating primitive markup.
- Never fork a primitive into a second near-duplicate file (`button.tsx` + `cta-button.tsx` with 90% overlap) — extend the variants instead.

## Variant naming (class-variance-authority)

Follow the pattern already established by the CLI-generated `Button` component in `components/ui/button.tsx`:
- `variant` for visual style (`default`, `secondary`, `outline`, `ghost`, `destructive`, plus any this template adds like `link`)
- `size` for dimensions (`default`, `sm`, `lg`, `icon`)
- Default both via `defaultVariants` in the `cva()` call so consumers can omit props entirely.

When adding a variant, check whether it's actually a new visual style or just a one-off className override at the call site — most one-off cases don't need a new named variant, just a `className` prop merged via `cn()`.
